36-hmr-esm

hmr esm

Understanding this sample

Standalone HTTP application

This sample starts a normal web server. External clients reach it via URLs (browser, curl, mobile app).

When this shape is used: Best when you expose a REST/JSON API, MVC pages, Swagger, or file uploads from one deployable Node app.

Bootstrap
NestFactory.create(AppModule)
Inventory kind
HTTP server

Default port: 3000

Global setup in main.ts: `ValidationPipe` (global, `@nestjs/common`)

In the inventory below, Wired (6) means Nest actively loads that class; Example (5) means the file is reference-only until you register it in a module or decorator.

Related concepts: application type explained · Wired vs Example · Why decorators? · Networking acronyms (TCP, RPC…)

36-hmr-esm — NestJS Sample

Webpack Hot Module Replacement with an ESM-oriented project layout ("type": "module"), plus the same cats-app patterns as 01-cats-app (guards, interceptors, pipes).

Quick start

cd sample/36-hmr-esm
npm install
npm run start:dev    # webpack HMR watch

App listens on http://localhost:3000 (or PORT env var).

Method Path Description
GET /cats List cats ({ data: [...] } interceptor)
POST /cats Create cat (@Roles(['admin']))
GET /cats/:id Stub; custom ParseIntPipe on :id

Project structure

sample/36-hmr-esm/
├── src/
│   ├── main.ts                       # HMR + global ValidationPipe
│   ├── app.module.ts
│   ├── cats/                         # Same layout as 01-cats-app
│   ├── core/                         # Global interceptors (active)
│   └── common/                       # Guards, pipes, etc. (mixed)
├── webpack-hmr.config.cjs
└── package.json                      # "type": "module"

Compare with 01-cats-app (same architecture) and 08-webpack (HMR only, no cats feature).

How the app boots

HMR in main.ts:

if (module.hot) {
  module.hot.accept();
  module.hot.dispose(() => app.close());
}
