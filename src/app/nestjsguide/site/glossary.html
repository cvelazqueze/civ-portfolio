NestJS building blocks

NestJS

NestJS is a Node.js framework for building server-side applications. It organizes code into modules, routes HTTP (or other transports) through controllers, and puts business logic in injectable providers (often called services).

It heavily uses TypeScript decorators (@Module(), @Controller(), @Get()…) so the framework can discover and wire classes automatically instead of you manually registering every route and dependency.

Decorator

A decorator is a function prefixed with @ that attaches metadata to a class, method, or parameter. Nest reads that metadata at startup to know:

  • which class is a controller and which URL prefix it owns (@Controller('cats'));
  • which method handles GET /cats (@Get());
  • what to inject in the constructor (@Injectable() enables DI);
  • which guards, pipes, or interceptors apply (@UseGuards(), etc.).

Why decorators? They keep routing and cross-cutting rules declarative and next to the code they affect, instead of scattered in a central router file. The same class reads as “this is a cats API” and Nest builds the HTTP surface from it.

In inventory tables, decorators are grouped by library: @nestjs/common (framework), user-created (your metadata, e.g. @Roles()), or third-party (e.g. class-validator).

Wired vs Example

Many samples include extra files under common/ to show a pattern without turning every demo into a huge app. The inventory marks each element:

  • Wired — registered in a module (providers, controllers, imports), applied globally in main.ts, or attached with @UseGuards() / similar. Nest actually runs this code.
  • Example (not registered) — source file exists for you to read and copy, but nothing imports or applies it yet. It is not active in the running app.

When learning, start with wired pieces — they define real behavior. Then open example files to see how you would add filters, middleware, or extra interceptors later.

Module

A module (@Module({...})) is a box that groups related controllers, providers, and imports. AppModule is the root; feature modules like CatsModule keep the app maintainable.

imports pulls in other modules. providers registers injectable classes. controllers registers HTTP/WebSocket/GraphQL entry points. exports makes providers available to modules that import this one.

Controller

Controllers handle incoming requests. In HTTP apps they define routes (@Get(), @Post()…) and call services. In microservice apps the same class may use @MessagePattern() or @EventPattern() instead of HTTP verbs.

Controllers should stay thin: validate input (often via pipes), delegate to a service, return the result.

Provider / Service

Providers are classes marked @Injectable(). Nest creates one shared instance (by default) and injects it wherever the constructor asks for it — that is dependency injection (DI).

Services are providers that hold business logic and data access. Controllers depend on services; services can depend on other services or repositories.

DI — Dependency injection

DI means Nest creates and connects your classes for you. Instead of new CatsService() inside a controller, you write constructor(private catsService: CatsService).

Nest’s container builds the graph, respects module boundaries, and makes testing easier (swap real services for mocks).

DTO — Data Transfer Object

A DTO is a simple class that describes the shape of data going in or out — for example “a new cat has a name string and an age number.”

Combined with class-validator decorators (@IsString(), …) and ValidationPipe, DTOs enforce input rules before your handler runs.

Guard

Guards run before the route handler. They answer: “should this request proceed?” (canActivate). Authentication and authorization live here — e.g. RolesGuard checking JWT roles.

Applied with @UseGuards() on a class or method, or globally in main.ts.

canActivate return value:

  • true — allow; Nest continues to pipes and the handler.
  • false — deny; Nest stops and sends 403 Forbidden (by default).

In sample 01-cats-app, if a route has no @Roles(...) metadata, RolesGuard returns true (“no role rule on this route, pass through”). Only routes you mark with @Roles are checked. See RolesGuard walkthrough and custom @Roles decorator.

Custom decorator + Reflector (e.g. @Roles)

Not every @Something comes from Nest. You can define your own decorator that attaches **metadata** (extra labels) to a class or method. Sample 01-cats-app defines @Roles in one small file:

export const Roles = Reflector.createDecorator<string[]>();

That file is a **blueprint** (factory): it does not contain 'admin'. When you write @Roles(['admin']) on create(), you **stamp** that route with “required roles = admin”.

Why Reflector from @nestjs/core? Nest’s core stores decorator metadata at startup. A guard injects Reflector and reads it back with reflector.get(Roles, handler). You use the framework’s metadata system instead of inventing your own.

User-created vs Nest built-in: @Get() is Nest; @Roles() is yours, but it follows the same metadata + guard pattern Nest recommends.

RolesGuard logic (canActivate explained)

From 01-cats-app — what the guard does in plain language:

  1. reflector.get(Roles, handler) — “What roles did this route require?” ['admin'] on POST /cats; nothing on GET /cats.
  2. if (!roles) return true — No @Roles on this route → allow. Protection is opt-in per handler.
  3. request.user — Who is calling; their roles should live on user.roles. (This sample does not set user; auth would, e.g. JWT sample 19.)
  4. user.roles.some(role => !!roles.find(item => item === role)) — “Does the user have at least one required role?”

Why !! before roles.find(...)? find returns the matched string (truthy) or undefined (falsy). !! converts that to a strict boolean true/false — a common JavaScript habit inside .some().

Why return true when there are no required roles? canActivate === true means “let the request through.” Routes without @Roles are not role-protected by this guard.

Full line-by-line table on the 01-cats-app sample page.

Pipe

Pipes transform or validate input (route params, query, body) before it reaches the handler. ValidationPipe runs class-validator on DTOs; custom pipes can parse :id strings into integers.

Interceptor

Interceptors wrap the handler’s observable/stream. They can log, measure time, or transform the output (e.g. wrap every response in { data: … }). They run after guards and pipes, around the actual business logic.

Exception filter

Filters catch thrown exceptions and format the HTTP/RPC error response. Without a filter, Nest uses defaults; custom filters unify error JSON or map domain errors to status codes.

Middleware

Express/Fastify-style functions that run for matching routes before guards. Useful for request logging or generic headers. Registered in a module’s configure(consumer) method.

HTTP server (standalone app)

Bootstrap: NestFactory.create(AppModule) then app.listen(port).

This is the most common shape: a single Node process exposes REST (or similar) endpoints. Clients call URLs; Nest routes to controllers. Samples like 01-cats-app and 10-fastify are standalone HTTP apps.

Why standalone? One deployable unit, simple local dev (browser or curl), ideal for CRUD APIs and monoliths.

Microservice only

Bootstrap: NestFactory.createMicroservice(AppModule, { transport: … }) — no public HTTP API port.

The app listens on a transport (TCP, gRPC, NATS, Redis, MQTT, …). Other services call it with messages instead of opening http://… URLs. See TCP, RPC, and Transport in this glossary.

Why? Split a big system into smaller deployable pieces, use efficient protocols, or plug into existing message infrastructure.

Hybrid HTTP + microservice

Bootstrap: NestFactory.create(AppModule), then app.connectMicroservice(...) and app.startAllMicroservices(), then app.listen().

One process runs both an HTTP server and a message listener. Sample 03-microservices uses this: HTTP GET / triggers an internal TCP RPC handled in the same codebase.

Why? Gradual migration, admin HTTP endpoints beside workers, or demos that show client and server without two repos.

Application context (no HTTP)

Bootstrap: NestFactory.createApplicationContext(AppModule) — no network listener.

Nest still builds the DI container and runs providers. Use this for CLI tools, cron scripts, queue consumers started manually, or integration tests that need real modules without opening a port.

Sample 18-context demonstrates fetching a service and calling methods, then closing the app.

WebSocket gateway

@WebSocketGateway() classes handle persistent socket connections (Socket.IO or native WebSocket). Methods use @SubscribeMessage() instead of @Get(). Used for chat, live updates, and games. See WebSocket and Socket.IO.

MessagePattern & EventPattern

@MessagePattern('cmd') — request/response over a microservice transport: send a message, wait for a reply (like a remote function call). See RPC.

@EventPattern('event') — fire-and-forget: publish an event, no reply expected. Common with pub/sub systems like NATS or Redis.

Web & API terms

HTTP — HyperText Transfer Protocol

HTTP is the language browsers and most APIs use on the web. A client sends a request (“GET this URL”) and the server sends a response (status code + body).

Common verbs: GET (read), POST (create), PUT/PATCH (update), DELETE (remove). Nest maps these with @Get(), @Post(), etc.

When a sample says it “listens on port 3000,” it means an HTTP server is waiting for those requests.

API — Application Programming Interface

An API is a defined way for one program to use another. A REST API exposes URLs your front-end or mobile app can call. An internal API might only be used by your own microservices.

In these samples, “the cats API” usually means the set of HTTP routes under /cats that create and list cats.

REST

REST is a popular style for HTTP APIs: each resource (users, cats, orders) has URLs, and you use HTTP verbs on them. Example: GET /cats lists cats, POST /cats creates one.

It is not a strict standard but a set of conventions Nest controllers follow naturally.

JSON — JavaScript Object Notation

JSON is a text format for structured data, easy for JavaScript and most languages to read:

{ "name": "Whiskers", "age": 3 }

Most Nest HTTP APIs send and receive JSON in request/response bodies unless the sample serves HTML or files instead.

CRUD

CRUD stands for the four basic database operations:

  • Create — add a new record
  • Read — fetch one or many
  • Update — change an existing record
  • Delete — remove a record

Many samples are thin CRUD APIs over a database or in-memory store.

MVC — Model, View, Controller

MVC splits an app into:

  • Model — data and business rules
  • View — what the user sees (HTML templates)
  • Controller — handles the request and picks a view

Samples 15-mvc and 17-mvc-fastify render HTML with Handlebars instead of returning JSON.

OpenAPI & Swagger

OpenAPI (formerly Swagger spec) is a standard way to describe REST APIs: paths, parameters, response shapes. Tools can generate documentation and client code from it.

Swagger UI is a web page that lists your routes and lets you try them in the browser. Sample 11-swagger auto-generates this from Nest decorators.

CORS — Cross-Origin Resource Sharing

CORS is a browser security rule. If your front-end runs on http://localhost:5173 but your API is on http://localhost:3000, the browser blocks requests unless the API explicitly allows that origin.

When you see CORS mentioned in server config, it means “which front-end URLs may call this API from a browser.”

SSE — Server-Sent Events

SSE lets the server push a stream of updates to the browser over a single long-lived HTTP connection — one direction only (server → client).

Lighter than WebSockets when you only need server push (live logs, stock ticks). Sample 28-sse uses Nest’s @Sse() decorator.

Networking, RPC & messaging

TCP — Transmission Control Protocol

TCP is a low-level way two programs on a network maintain a reliable connection and exchange bytes. It sits below HTTP — HTTP actually runs on top of TCP.

In sample 03-microservices, Nest’s Transport.TCP means services send structured messages over a TCP socket instead of REST URLs. You don’t browse to “tcp://…” in a browser; another Nest process (or client) connects programmatically.

Think of TCP as a phone line; the app defines what language you speak on that line (Nest’s message format).

RPC — Remote Procedure Call

RPC means calling a function that runs on another machine (or another process) as if it were local. You send “run sum with [1,2,3]”; the remote side executes and sends back 6.

In Nest, @MessagePattern({ cmd: 'sum' }) marks the handler for such a call. Sample 03-microservices demonstrates HTTP triggering an internal TCP RPC.

RPC is the idea; TCP, gRPC, and Redis are different ways to carry RPC messages.

gRPC — Google Remote Procedure Call

gRPC is Google’s modern RPC system. It uses Protocol Buffers (binary contracts defined in .proto files) and HTTP/2 under the hood — fast and strongly typed.

Sample 04-grpc is a Nest microservice with no REST routes; handlers use @GrpcMethod() instead of @Get().

Use gRPC when services need efficient internal communication and shared schemas, not when humans need to “open a URL in Chrome.”

Transport (Nest microservices)

In Nest, a transport is the wire protocol your microservice listens on. Examples:

  • TCP — direct socket connection (sample 03)
  • GRPC — gRPC protocol (sample 04)
  • REDIS — messages via Redis pub/sub
  • NATS, MQTT, Kafka — message brokers (integration tests in this repo)

You choose transport in createMicroservice({ transport: Transport.TCP, … }) or connectMicroservice(...).

WebSocket

A WebSocket is a persistent, two-way connection between browser and server. Either side can send messages anytime — unlike HTTP where the client always initiates one request/response.

Used for chat, multiplayer games, live dashboards. Nest exposes them via @WebSocketGateway() (samples 02-gateways, 16-gateways-ws).

Socket.IO

Socket.IO is a library on top of WebSockets with fallbacks and rooms/namespaces. Sample 02-gateways uses the default Socket.IO adapter.

Sample 16-gateways-ws uses the native WebSocket driver instead — fewer features, less overhead.

Pub/sub — Publish / subscribe

Pub/sub is a messaging pattern: publishers send events to a channel; subscribers listen without knowing who sent it.

Example: when an order is placed, one service publishes order.created; email and inventory services each subscribe and react.

NATS, Redis, MQTT, and @nestjs/event-emitter (in-process) all use variations of this idea.

NATS

NATS is a lightweight, high-performance message broker — a server that routes pub/sub and request/reply messages between apps.

Common in microservice architectures when you want decoupled services without HTTP between every pair. Nest supports NATS as a microservice transport (see integration tests in this repository).

MQTT

MQTT is a minimal pub/sub protocol popular in IoT (sensors, devices on slow networks).

Messages are published to topics (like sensors/temperature). Nest can connect as an MQTT microservice transport when devices or brokers speak MQTT.

Redis

Redis is an in-memory data store, often used for:

  • Caching — store API responses temporarily (sample 20-cache may use cache-manager with Redis)
  • Job queues — Bull/BullMQ store jobs in Redis (sample 26-queues)
  • Pub/sub — Nest microservice transport over Redis channels

Fast because data lives in RAM; usually backed by disk persistence options for durability.

Databases & ORMs

SQL

SQL (Structured Query Language) is the standard language for talking to relational databases: tables with rows and columns, joins, transactions.

Examples: PostgreSQL, MySQL, SQLite. Samples 05-sql-typeorm and 07-sequelize use SQL databases.

ORM — Object-Relational Mapping

An ORM lets you work with database rows as TypeScript classes instead of writing raw SQL everywhere.

You define an User entity class; the ORM generates queries to insert and fetch users. Trade-off: convenience vs fine-grained SQL control.

TypeORM

TypeORM is a TypeScript ORM integrated with Nest via @nestjs/typeorm. Entities use decorators like @Entity(), @Column().

Used in samples 05-sql-typeorm (PostgreSQL) and 13-mongo-typeorm (MongoDB via TypeORM’s document support).

Sequelize

Sequelize is another Node ORM with a model-based API. Sample 07-sequelize shows Nest’s @nestjs/sequelize integration.

Similar goal to TypeORM; teams often pick one ORM and stick with it across a project.

Mongoose

Mongoose is the main ODM (Object Document Mapper) for MongoDB in Node. Schemas define flexible JSON-like documents.

Samples 06-mongoose and 14-mongoose-base store data in MongoDB collections instead of SQL tables.

Prisma

Prisma is a modern database toolkit: schema in schema.prisma, generated type-safe client, migrations. Sample 22-graphql-prisma uses Prisma with GraphQL resolvers.

Often chosen for strong TypeScript autocomplete and clear migration workflow.

MongoDB

MongoDB is a document database — data is stored as flexible JSON-like documents in collections, not fixed SQL tables.

Good for nested data and rapid schema changes. Accessed in these samples via Mongoose or TypeORM.

PostgreSQL

PostgreSQL (often “Postgres”) is a powerful open-source relational SQL database. Sample 05-sql-typeorm connects Nest to Postgres for typed entities and relations.

SQLite

SQLite is a small SQL database stored in a single file on disk — no separate database server to install.

Sample 22-graphql-prisma uses SQLite for easy local dev (dev.db).

GraphQL & federation

GraphQL

GraphQL is an API style where the client sends a query describing exactly which fields it wants ({ user { name email } }), and the server returns JSON matching that shape.

One endpoint (often /graphql) replaces many REST URLs. Nest supports GraphQL via @nestjs/graphql with code-first or schema-first setup.

GraphQL resolver

A resolver is the GraphQL equivalent of a controller. @Resolver() classes define @Query() (read), @Mutation() (write), and @Subscription() (live updates) handlers.

SDL — Schema Definition Language

SDL is the text syntax for GraphQL schemas:

type User {
  id: ID!
  name: String!
}

Schema-first samples (e.g. 12-graphql-schema-first) write .graphql files first, then generate TypeScript types. Code-first builds the schema from TypeScript decorators instead.

GraphQL federation

Federation splits one GraphQL API across multiple backend services. Each service owns part of the graph; a gateway merges them so clients still see a single endpoint.

Samples 31 and 32 run separate apps for users, posts, and gateway — closer to how large companies structure GraphQL at scale.

Subgraph

A subgraph is one service’s slice of a federated GraphQL API. The users app might own the User type; the posts app owns Post.

Each subgraph runs its own Nest app with /graphql; the gateway stitches them together.

Apollo

Apollo is a popular GraphQL ecosystem (server libraries, federation tools, client). Many Nest GraphQL samples use Apollo-compatible drivers and federation directives like @key for entity identity.

Mercurius

Mercurius is a GraphQL server plugin for Fastify — an alternative to the default Express-based GraphQL setup. Sample 33-graphql-mercurius combines Fastify + Mercurius for performance.

Tools, auth & build

Node.js

Node.js runs JavaScript and TypeScript outside the browser — on servers, CLIs, and build tools. NestJS is built on Node.

You install Node (which includes npm) to run npm install and npm run start:dev in each sample.

TypeScript

TypeScript is JavaScript with optional static types (name: string). Nest samples are mostly TypeScript; it catches mistakes before runtime and powers decorator metadata.

Sample 09-babel-example shows the same ideas in plain JavaScript with Babel.

See also: Classes, interfaces & types — what they are and why Nest uses all three.

Classes, interfaces & types (TypeScript)

Nest apps are TypeScript. Three shapes show up everywhere — they look similar but do different jobs.

Class

A class is a blueprint for objects that exist at runtime. It can have:

  • Properties — data on the instance (name: string)
  • Methods — functions on the instance (findAll())
  • Decorators — @Injectable(), @Controller(), @IsString() (Nest/class-validator read these when the app starts)

In Nest, almost everything “alive” is a class: controllers, services, guards, DTOs. Example from 01-cats-app:

export class CatsService {
  private readonly cats: Cat[] = [];
  findAll(): Promise<Cat[]> { ... }
}

Why classes here? Nest’s DI container instantiates classes, injects dependencies into constructors, and reads decorator metadata from the class prototype.

Interface

An interface describes the shape of an object — field names and types — but does not exist at runtime. TypeScript removes it when compiling to JavaScript.

export interface Cat {
  name: string;
  age: number;
  breed: string;
}

Why interfaces? Document “what a cat object looks like” without creating a class. CatsService stores Cat[] and returns Promise<Cat[]> so editors and the compiler catch typos (naem instead of name).

Interfaces cannot have decorators or methods with bodies in the usual Nest DTO sense — they are type-only.

Type (type alias)

A type (from type X = …) also describes shapes, often for unions, primitives, or aliases:

type Role = 'admin' | 'user';
type CatId = number;
type CatList = Cat[];

Interface vs type — practical difference:

  • Interface — best for object shapes you may extend (interface Admin extends User). Common for domain models like Cat.
  • Type alias — best for unions, tuples, mapped types, or “alias this complex shape”. Either works for simple object shapes; teams pick one style.

Both are erased at runtime — only TypeScript checks them while you code.

Side-by-side in 01-cats-app

Why DTO is a class but Cat is an interface

CreateCatDto must be a class because class-validator decorators (@IsString()) attach to the class at runtime and ValidationPipe inspects them.

Cat is only used for typing inside your code — no validation decorators needed — so a lightweight interface is enough.

Why any of this matters in Nest

  • Fewer bugs — wrong property names or types fail at compile time.
  • Clear contracts — controllers, services, and DTOs document what flows through the API.
  • Framework features — DI, validation, and Swagger often require classes with decorators; interfaces/types support the type checker only.
  • Readable samples — seeing Cat vs CreateCatDto teaches input (request) vs domain (your app’s model) separation.

npm

npm (Node Package Manager) installs open-source libraries listed in package.json. npm install downloads dependencies; npm run … runs scripts like starting the dev server.

ESM — ECMAScript Modules

ESM is the modern JavaScript module system: import and export syntax. Older Node code often used CommonJS (require).

Samples 34, 35, and 36 deal with ESM-only npm packages and hot reload — relevant as the Node ecosystem shifts toward ESM.

CLI — Command Line Interface

A CLI is a program you run in a terminal (no graphical UI). The Nest CLI (nest generate) scaffolds modules and controllers.

Sample 18-context uses an application context suitable for CLI scripts that need services without HTTP.

Express

Express is the default HTTP engine under many Nest apps — a minimal Node web framework. Nest adds structure (modules, DI) on top.

Sample 15-mvc uses Express-specific view APIs via NestExpressApplication.

Fastify

Fastify is an alternative HTTP engine focused on speed and low overhead. Nest supports it via NestFactory.create(..., new FastifyAdapter()).

Samples 10-fastify, 17-mvc-fastify, and 33-graphql-mercurius use Fastify instead of Express.

JWT — JSON Web Token

A JWT is a compact, signed string that proves “this user logged in successfully.” The server creates it after login; the client sends it on later requests (often in an Authorization: Bearer … header).

Sample 19-auth-jwt uses JWT with Passport strategies and guards — stateless auth without server-side sessions.

Passport

Passport is a Node authentication library with “strategies” (local username/password, JWT, OAuth, etc.). Nest wraps it in @nestjs/passport and connects strategies to guards.

Bull (job queue)

Bull is a Redis-backed job queue for Node. You enqueue work (e.g. “transcode this audio”); a processor picks it up asynchronously.

Sample 26-queues uses @Processor() and @Process() — HTTP triggers jobs, workers handle them in the background.

Multer (file uploads)

Multer handles multipart/form-data — how browsers upload files in forms. Nest integrates it via @UploadedFile() and FileInterceptor.

Sample 29-file-upload demonstrates receiving files over HTTP.

Cron (scheduled tasks)

Cron jobs run on a schedule (“every day at midnight”, “every 5 seconds”). Named after the Unix cron scheduler.

Sample 27-scheduling uses @Cron(), @Interval(), and @Timeout() from @nestjs/schedule.

Webpack

Webpack bundles many source files into fewer outputs for deployment or dev. Sample 08-webpack uses Webpack with hot reload instead of plain tsc during development.

Babel

Babel transpiles modern (or experimental) JavaScript into something Node understands — including decorators in sample 09-babel-example, which runs Nest without TypeScript.

HMR — Hot Module Replacement

HMR swaps changed code into a running app without a full restart — faster feedback while coding. Sample 36-hmr-esm and 08-webpack demonstrate HMR with Nest.

Development convenience only; production builds still use normal compiles.

Handlebars

Handlebars is an HTML template engine — you write .hbs files with placeholders; the server fills them with data and returns HTML.

Used in MVC samples for server-rendered pages instead of JSON APIs.

YAML

YAML is a human-readable config format (indentation-based), often used for Docker Compose, CI configs, and OpenAPI files. Easier to read than JSON for long configuration.
